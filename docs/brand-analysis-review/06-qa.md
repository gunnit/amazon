# Brand Analysis & Brand Pulse — QA / Test Strategy & Regression Safety Review

**Agent:** QA Lead
**Scope:** Test coverage audit + regression-safety plan for the full rework (terminology/flow, remove/cancel analysis, 10 new data sources, dynamic PPTX, Brand Pulse → Weekly Brand Intelligence Report, in-app notifications).
**Branch:** `master` @ `156536c`. All `file:line` anchors verified against working tree.

---

## 0. TL;DR for the steering group

- **The current test suite is a thin shell of pure-function unit tests with hand-rolled fakes.** There is **no `conftest.py`, no DB integration, no API/endpoint test, no async-job/processor test, no Brand Pulse test, no capabilities-persistence test, no storage test, and no notification test for Brand Analysis.** The four in-scope test files (`test_brand_analysis_service.py` 668 lines, `test_brand_analysis_sources.py` 362, `test_market_research_pricing.py`, `test_market_research_comparison.py`) cover parse/metrics/provenance/PPTX-structure and source-adapter filtering only.
- **The single most dangerous brittleness is the hard-coded slide count.** Three separate places assert exact deck size: `test_pptx_generation_produces_valid_deck` asserts `== 15` (`test_brand_analysis_service.py:207`), the pipeline test asserts `== 15` (`:280`), and `validate_pptx_bytes` asserts the range `12–16` (`brand_analysis_service.py:3198`). **The PPTX rework's entire premise is dynamic slide count.** Every one of these will fail on the first redesign commit and there is no contract test that survives the transition. This is the #1 thing to fix *before* the rework starts, not after.
- **Remove/Cancel has a proven data-integrity bug today.** `delete_job` (`brand_analysis_service.py:3097`) does a bare `db.delete(job)` with **no status guard**, and the processor's success path (`:3553-3572`) commits a fully-populated `job` object **without re-checking the row still exists** — the discovery map's "resurrect a deleted row" risk is real and exploitable by a delete-during-processing race. There is **zero** Celery `revoke`/`AsyncResult`/`task_id` plumbing anywhere in the codebase (grep confirmed), so "Cancel In Progress" cannot even be built on the current model without a new `celery_task_id` column and a cooperative cancel-check.
- **The `__DAILY_TOTAL__` double-count gotcha is NOT tested anywhere.** `AmazonAccountDataSource` excludes `DAILY_TOTAL_ASIN` (per discovery `sources.py`), and `BrandPulseService` delegates to `AnalyticsService.compute_dashboard_kpis` which *uses* the `DAILY_TOTAL` sentinel for totals while `asin_sales_breakdown` sums per-ASIN — these two number sources can disagree and there is no test pinning them. This is the highest-severity untested data-integrity risk for both modules.
- **In-app notifications: good news.** A mature read-state pattern already exists in `app/api/v1/alerts.py` (`unread_count` at `:84/:138`, `is_read` at `:100/:416`, status-vs-is_read reconciliation `:336-351`, partial index `ix_alerts_rule_unread_triggered_at` in `alert.py:55-60`). But it is **rule/event-scoped, not a generic user-facing notification store** — a new `notifications` table + endpoints are needed; the alerts code is the template to copy, not reuse.

**Verdict:** The test strategy needs a **rebuild, not an improvement.** Current tests are fine as far as they go but they test the *easy* deterministic core and leave every risky seam (async lifecycle, DB, API, SP-API integration, scheduler, LLM output) completely unguarded. Ship the rework behind a new test pyramid or it will regress silently.

### 1.2 The test infrastructure is primitive

- **No `conftest.py`** (confirmed: `tests/conftest.py` does not exist). No shared fixtures, no DB fixture, no FastAPI `TestClient`/`ASGITransport` harness. `pytest-asyncio==0.23.3` is installed (`requirements.txt:59`) but **not used** — every async test calls a hand-rolled `_run(coro)` = `asyncio.run(coro)` (`test_brand_analysis_service.py:138`, `test_brand_analysis_sources.py:80`).
- **No API-level tests** anywhere for brand analysis/pulse. The only `TestClient`-style tests in the repo are `test_request_id_middleware.py` / `test_validation_exception_handler.py`. So `create_job` 422 paths, the `409` already-running guard (`brand_analysis.py:235`), the manual-mode "both years required" guard (`:242`), the download 422s (`:292`), and delete are **never exercised**.
- **Fakes are duplicated and drift-prone.** `_FakeDb` (`test_brand_analysis_service.py:537`), `FakeDb` (`test_brand_analysis_sources.py:44`), `_FakeCapabilityClient` (`:584`), and `FakeClient` variants in market-research tests all re-implement SQLAlchemy `execute`/result semantics by hand and string-match SQL (`"FROM products" in statement_text`). A query refactor that changes SQL text silently invalidates the fakes without failing a test.

### 1.3 PPTX tests are pinned to today's static deck — incompatible with the rework

- `test_pptx_generation_produces_valid_deck` (`:197`) asserts **exactly 15 slides** with a comment hard-coding the reasoning ("the market share slide is skipped and the deck has 15 slides").
- `test_pptx_includes_market_share_slide_when_external_market_export_present` (`:228`) asserts `na_count + 1` — the *only* test that checks a slide appears/disappears with data. It is the seed of the right pattern, but it mutates `metrics["market_analysis"]` in place and assumes the rest of the deck is fixed.
- `validate_pptx_bytes` asserts `12 <= slide_count <= 16` (`:3198`) and is called inside the processor (per discovery `:3534`) as a **hard gate** — a dynamic deck that legitimately produces 8 or 20 slides will **fail the job in production**, not just the test.
- Tests assert literal copy strings: `"CURRENT AMAZON PERFORMANCE"` (`:221`), `"SEO & CONTENT"` (`:225`), `"MARKET SHARE"` presence/absence (`:224,:244`). Every redesigned title breaks these.

### 1.4 Remove / Cancel — current code is unsafe and untestable

- `delete_job` (`:3097-3103`): `get_job` → `db.delete(job)` → `flush`. **No check on `job.status`.** Deleting a `generating_pptx` job removes the row while the processor thread/worker still holds a live ORM object.
- The processor success path (`:3553-3572`) sets `job.metrics = ...` etc. on an object loaded earlier in `_process()` and calls `await db.commit()` **without re-selecting to confirm the row still exists** → SQLAlchemy re-INSERTs/UPDATEs a deleted PK, resurrecting it. The failure path (`:3576-3586`) *does* re-select (`select(BrandAnalysisJob).where(id==...)`) so it is safe; the **success path is not.**
- **No cancellation primitive exists.** Grep for `revoke|AsyncResult|task_id|celery_task_id` across `app/` returns nothing in brand-analysis scope. Celery task is `max_retries=1, countdown=60` (`workers/tasks/brand_analysis.py`), and the in-process thread fallback (`brand_analysis.py:269-274`) is fire-and-forget and **un-revocable by construction**.
- **No stuck-job recovery.** `next_retry_at` / `sync_attempt_count` columns exist (migration 021) but nothing polls them — unlike `recover_stuck_scheduled_report_runs` (`scheduled_reports.py:96-137`), which is the exact pattern to copy.

### 1.5 Brand Pulse — zero tests, real overlap, copied constants

- **No test file** references `BrandPulseService` or `brand_pulse`. Confirmed by grep.
- `DECLINE_THRESHOLD_PCT = -5.0` / `DECLINE_FAST_THRESHOLD_PCT = -20.0` (`brand_pulse_service.py:25-26`) are **copied** from `AnalyticsService` with a comment admitting it ("Mirror AnalyticsService's trend thresholds"). Nothing pins them equal → silent drift.
- Numbers come from two different primitives that can disagree: `compute_dashboard_kpis` (`:49`, uses `DAILY_TOTAL` sentinel) for the overview totals vs `asin_sales_breakdown` (`:63`, per-ASIN sum) for top/declining. **No test asserts `sum(top_asins.revenue) ≤ overview.revenue`** or that the sentinel isn't double-counted.
- `_ads_block` TACOS = `spend / total_revenue * 100` (`:180`) with a `total_revenue > 0` guard but **no test** for the divide-by-zero / vendor-monthly-lag `awaiting_data` branch (`:57-62`).

### 1.6 In-app notifications — pattern exists, store does not

- `notification_service.py` is **email/webhook only** (SendGrid + httpx). The MEMORY blocker confirms SendGrid is dead in prod. Brand-analysis-complete must be **in-app only** (per requirements) — so `NotificationService` is the wrong tool.
- The right template is `alerts.py`: `unread_count` (`:84,:138,:143`), `is_read` toggling (`:100,:416`), `status`-vs-`is_read` reconciliation (`:336-351`), and the `Alert` model's partial unread index (`alert.py:55-60`). But `Alert` is `rule_id`-scoped (FK to `alert_rules`, `alert.py:64`) — it cannot represent a "your Brand Analysis is ready" event without a rule. **A new generic table is required.**

### 1.7 The 10 new data sources — capabilities are probed but consumption is untested

- `test_capability_detection_*` (`:629-667`) test the **probe** (happy + 403 paths) via `_FakeCapabilityClient` with per-API exception injection. Good. But:
  - The probe persists 11 capabilities (`CAPABILITY_KEYS`), of which **5+ (`data_kiosk`, `brand_analytics`, `finance`, `settlement`, `listings`) are never consumed by metric calc** (discovery §4). There is no test asserting "capability present ⇒ metric appears," because that wiring doesn't exist yet.
  - `_FakeCapabilityClient` returns canned `payload={}` shapes — it does **not** model real SP-API response envelopes, pagination, or throttling (429). When the rework actually *integrates* these sources, the fakes are insufficient.

### 4.8 Acceptance criteria & release-gating regression checklist

**Acceptance criteria (per area):**
- Remove/Cancel: cannot delete a running job (409); cancel terminates within one status checkpoint; no resurrected rows; artifact + source files + snapshots GC'd on delete.
- Dynamic PPTX: zero empty-placeholder sections in any data scenario; deck opens as OOXML; manifest matches data presence; no hard slide-count assertion remains in the suite.

## 6. Risks (and mitigations)

| Risk | Mitigation |
|---|---|
| **Slide-count tests block the PPTX rework or, worse, are loosened without replacing the contract** → empty placeholders ship silently. | Land R2 (manifest + empty-placeholder assertion) **before** deleting count assertions; make "no empty section" a release gate. |
| **DB-integration fixture is slow/flaky in CI** (testcontainers, asyncpg cross-loop). | Reuse the proven `reset_engine_for_worker` + fresh-loop pattern (`scheduled_reports.py:17`); transactional rollback per test; keep heavy E2E + visual-diff on nightly, not per-PR. |
| **Delete/cancel race fix is incomplete** — fixing `delete_job` but not the success-path re-select still resurrects rows. | Test the race explicitly (R3 `test_processor_does_not_resurrect_deleted_job`); require both the guard AND the re-select to land together. |
| **New-source mocks diverge from real SP-API** → green tests, broken prod (the exact thing that produced the original "sources not obtained" finding). | Capture one real (sanitized) envelope per source as a golden fixture; add a contract test that the mock shape matches the golden keys. |
| **LLM guardrail tests are non-deterministic** if they call the real model. | Always mock Anthropic in unit tests (the suite already does via `api_key=None`, `test_…:191`); test the *validator* and *fallback*, never live generations. |
| **Brand Pulse repositioning lands before its tests** (zero coverage today). | Gate the WoW-diff + `__DAILY_TOTAL__` + LLM-schema tests as P0 for that module; do not merge the reposition without them. |
| **Fakes keep drifting** as queries are refactored during the monolith split. | Migrate off SQL-string-matching fakes to the real DB fixture (R1); delete `_FakeDb`/`FakeDb` as each test moves. |
| **In-app notification table conflated with `Alert`** (tempting reuse) → cross-tenant leak or schema contortion. | Build a dedicated `notifications` table; reuse only the *read-state pattern* from `alerts.py`, not the `Alert` model itself; org-scoping test is a release gate. |

---

### Closing opinion

The deterministic core (parse → metrics → provenance) is genuinely well-tested and should be **preserved**. Everything *around* it — the async lifecycle, the DB, the API surface, the PPTX structure, Brand Pulse, and notifications — is **untested and is exactly what the rework touches.** Do not bolt new tests onto the existing `asyncio.run`-per-test style; that ceiling is why there are no integration tests today. Build the harness first (R1), pin the two data-integrity invariants that have already bitten this team (`__DAILY_TOTAL__` and delete-during-processing), and replace slide-count assertions with a section contract before a single deck is redesigned. Those four P0 items are the difference between a rework that ships safely and one that regresses in front of Gioia on PROD.
